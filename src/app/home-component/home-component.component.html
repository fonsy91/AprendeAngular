1. ESTRUCTURA Y MODIFICACION DE LA APP.

La diferencia entre una pagina web convencional y una single page application (SPA) estriba en que la primera esta haciendo peticiones al servidor continuamente cada vez que queremos ver una imagen se le hace una peticion al servidor y este nos devuelve otro html con esa imagen, en cambio para una SPA esto no es asi esta realiza una sola peticion al arranque de la aplicacion. El cliente va modificando cosas en la aplicacion y van apareciendo otras esto se hace gracias a la tecnologia AJAX es decir cada vez que el usuario realiza una accion se envia una peticion al servidor y este nos responde con un documento JSON se usa para intercambiar datos entre el cliente y el servidor podemos apreciarlo en la imagen de la izquierda.

5. BINDING BIDIRECCIONAL

Como vimos anteriormente en las explicaciones de la propiedad binding en ese caso cuando cambiabas una propiedad en el .ts esta se modificaba en el html esto implicaba una sola direccion, y en el event binding se hacia lo mismo pero atrapando el evento y a traves de una funcion cuando el usuario interactuaba con la aplicacion o el tempalte eso tenia reflejo en el .ts ejecutandose dicha funcion en este caso el flujo es contrario al del binding, cuando el flujo va tanto de un lado para otro se llama binding bidireccional o two way binding, esto nos permite actualizaciones continuas es decir modificar cosas en nel .ts a partir de accione en html y estas .ts se modifican al instante en el html otra vez. Para poder hacer todo esto necesitamos: la directiva ngModel y el uso de Banana in Box que es una sintaxis si recordais para el property banding usabamos [] para el flujo .ts hacia html y para el event binding () flujo de html a .ts, para este caso que queremos que sea bidireccional tendremos que suar ambos simbolos [()]. 9. SERVICIOS

Cuando una aplicacion angular esta compuesta de uno mas componentes y uno de ellos necesita hacer uso de unos datos que por ejemplo vienen de una base de datos y estos datos van a ser usados por uno mas componentes vamos a modular estos datos para que el componente que quiera usarlos pueda a esto se le llama servicio de esta manera no tienes porque crear un metodo en cada componente creas un servicio y sera proporcionado a cada componente que lo necesite, para ello crearemos un servicio hipotetico en el que al realizar un registro de una persona (practica de directivas) aparezca una ventana emergente mostrando nombre y apellidos y tambien cuando se agrege una caracteristica, se inyectara este servicio a dos de los 3 componentes el componente padre y el componente nieto.
Una vez creas el servicio ng g s "nombre_servicio" se agrega al app.module.ts en providers. La inyeccion del servicio siempre se hace a traves del constructor. En resumen es Básicamente un servicio es un proveedor de datos, que mantiene lógica de acceso a ellos y operativa relacionada con el negocio y las cosas que se hacen con los datos dentro de una aplicación. Los servicios serán consumidos por los componentes, que delegarán en ellos la responsabilidad de acceder a la información y la realización de operaciones con los datos. Como hemos dicho antes la inyeccion se hace desde el constructor y se pueden realizar las operaciones de ese servicio tanto desde el constructor como del metodo ngOnInit que es el segundo que se ejecuta una vez se arranca una aplicacion angular.
Tambien es posible agregar un servicio dentro de otro para ello hacemos uso de @Injectable() dentro del servicio donde queremos agregar otro servicio y en el constructor inyectar el otro servicio.
